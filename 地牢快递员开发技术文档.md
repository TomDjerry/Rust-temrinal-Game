TRD: 地牢快递员 (Dungeon Courier) - 技术设计文档
1. 项目概述 (Project Overview)
本旨在开发一款基于终端 (TUI) 的回合制 Roguelike 游戏。玩家将扮演一名地下城快递员，在充满敌意和随机生成的迷宫中，履行契约，将包裹安全送达。项目侧重于构建一个高扩展性、数据驱动的底层框架，为后续的复杂玩法（技能树、多层地牢、复杂 AI）提供工程基础。

2. 技术栈选型 (Technology Stack)
为了保证性能、可维护性和内存安全，本项目采用 Rust 作为核心开发语言，并采用以下技术栈：

UI 与渲染引擎:

ratatui: 现代化的 TUI 渲染库，负责组件化 UI 布局。

crossterm: 跨平台终端后端，处理输入监听、Raw Mode 切换和光标控制。

核心架构 (ECS):

hecs: 轻量级、高性能的 ECS（实体组件系统）库。强烈建议引入正式的 ECS 而非手写组件容器，这能极大地解耦“数据”与“逻辑”，是复杂 Roguelike 的行业标准。

地图与算法:

pathfinding: 提供现成的 A*、Dijkstra 等寻路算法，减少重复造轮子。

rand & rand_chacha: 提供核心随机数生成与确定性随机种子（ChaCha 算法性能好且跨平台一致）。

数据驱动与序列化:

serde + serde_json / ron (Rusty Object Notation): ron 非常适合用作 Rust 游戏的数据配置文件。

bincode: 用于实现极小体积的快速二进制存档。

日志与错误处理:

anyhow + thiserror: 规范化全局错误传播与模块级自定义错误。

tracing + tracing-appender: 替代简单的 log，提供结构化日志，方便在 TUI 后台异步写入日志文件进行 Debug。

3. 核心架构设计 (Architecture Design)
3.1 游戏主循环与状态机 (Game Loop & State Machine)
使用状态机模式管理游戏的宏观生命周期。主循环是受输入驱动的（Turn-based Blocking）。

App States: MainMenu -> GeneratingWorld -> InGame -> Inventory / QuestMenu -> GameOver。

Turn States (InGame): AwaitingInput -> PlayerTurn -> MonsterTurn -> EnvironmentTurn -> Render。

3.2 事件总线机制 (Event Bus System)
摒弃直接的方法调用，引入消息队列（基于 std::sync::mpsc 或自定义事件队列）。

目的: 彻底解耦系统。例如，攻击系统只需要发出 CombatEvent，而无需关心日志系统如何打印，或者噪音系统如何吸引怪物。

事件类型: MovementEvent, CombatEvent(Attacker, Defender, Damage), ItemPickupEvent, LogMessageEvent, NoiseEvent。

3.3 ECS 系统划分 (ECS Systems)
将逻辑拆分为独立的 System，按顺序在每个回合执行：

Input System: 将玩家按键转化为具体的意图（Intent Components）。

Movement System: 处理位移、碰撞检测（墙壁、实体拦截）。

Combat System: 消费攻击意图，计算伤害（结合属性、装备），应用状态效果。

AI System: 更新怪物状态机，计算寻路（BFS/A*），生成移动或攻击意图。

FOV System: 基于玩家的新位置，重新计算视线遮挡。

Death/Cleanup System: 处理 HP 为 0 的实体，掉落物品，清理内存。

4. 核心功能模块详细设计 (Subsystems)
4.1 过程生成地图系统 (Procedural Generation)
算法: BSP (二叉空间分割) 结合随机漫步算法生成房间和 L 型走廊。

确定性生成: 地图生成器必须接受一个 u64 的 Seed。所有环境实体的坐标、掉落物、怪物种类严格依赖此 Seed，确保 --seed 123 每次生成的副本完全一致。

Tile 数据结构:

Rust
pub struct Tile {
    pub tile_type: TileType,
    pub is_walkable: bool,
    pub is_opaque: bool,
    pub explored: bool,
}
4.2 视野与迷雾 (Shadowcasting FOV)
采用 Recursive Shadowcasting (递归阴影投射) 算法，这比射线检测更高效且没有视野盲区瑕疵。

渲染逻辑:

在 FOV 内：以明亮颜色（Foreground）渲染实体和地形。

曾探索但不在 FOV 内：过滤掉动态实体（怪物不显示），地形以暗灰色渲染。

未探索：不渲染（或渲染黑色）。

4.3 智能 AI 模块 (State-Driven AI)
怪物拥有 Brain 组件，内置状态机：

巡逻 (Patrol): 在出生点附近随机漫步或延既定路线移动。

警觉 (Investigate): 监听 NoiseEvent，如果噪音在听力范围内，将目标点设为噪音源并寻路。

追踪 (Chase): 当玩家进入怪物的 FOV，切换为追踪状态，使用 A* 算法向玩家移动。

Dijkstra Maps (进阶优化): 如果同屏怪物较多，可以构建以玩家为中心的 Dijkstra Map（热力图），所有怪物只需向“温度”更低的格子移动即可实现包围或追踪，极大降低 CPU 消耗。

4.4 数据驱动设计 (DOD)
所有非硬编码的逻辑必须提取到 assets/ 目录中。在游戏启动时，解析为 Registry 单例。

5. UI 与终端交互规范 (UI & Rendering Pipeline)
使用 ratatui 的 Layout 进行弹性切分。

根布局 (Root): 左右切分 (70% 游戏区, 30% 信息区)。

信息区布局 (Sidebar): 上下切分（角色面板、迷你任务面板、事件日志滚动区）。

渲染管线 (Render Pipeline):

清空上一帧缓冲区。

迭代 TileMap，绘制基础地图。

查询 ECS 中拥有 Renderable 与 Position 的实体，按 Z-index (Layer) 排序后绘制（地板物品 < 怪物 < 玩家）。

绘制侧边栏 UI 与边框（Block）。

渲染到终端并执行 flush()。

6. 开发里程碑与冲刺计划 (Milestones & Sprints)
为了保证项目的可执行性，采用敏捷开发的迭代思路：

Sprint 1: 引擎地基与渲染 (MVP 基础)

搭建 TUI 框架与 ECS 世界。

实现 BSP 地图生成算法。

实现玩家的输入与移动（带墙壁碰撞）。

交付物: 可以在终端里按 WASD 在随机生成的房间里走动。

Sprint 2: 核心玩法闭环 (Combat & Loop)

引入 Stats 组件和近战战斗公式。

加入呆板怪物（随机走动）和基础拾取物（回血药）。

实现全局事件日志（战斗伤害提示）。

实现包裹实体、出口实体与胜负判定。

交付物: 具备血量管理，能打死怪物，能走到出口通关。

Sprint 3: 战争迷雾与 AI (Tactics)

实装 FOV 算法。

实装怪物的状态机模型与 A* 寻路。

交付物: 怪物会在视野边缘发现你并追杀你，策略性大幅提升。

Sprint 4: 系统深化与外部数据 (Data-Driven)

接入 serde 和 ron，将物品、怪物配置抽离。

实现装备栏（武器、护甲）与背包 UI 弹窗。

契约系统（随机读取支线任务条件）。

交付物: 不写代码即可配置出上百种道具和怪物，支持复杂任务。

Sprint 5: 序列化与打磨 (Polish)

实装基于 bincode 的 ECS 状态存档/读档。

添加环境交互（开门、锁、陷阱、噪音）。

单元测试覆盖（寻路、生成、伤害公式）。

7. 风险评估与规避建议
终端刷新闪烁问题:

规避: 严格使用 crossterm 的 Alternate Screen 模式，并且只在每回合逻辑结算完毕后，整体执行一次 Terminal::draw，避免局部刷新带来的撕裂。

ECS 设计初期的心智负担:

规避: 刚开始不要过度拆分组件。例如 Stats 可以先包含在 CombatComponent 中。等到系统变得臃肿时，再将组件拆细。保持实体组件的扁平化。
