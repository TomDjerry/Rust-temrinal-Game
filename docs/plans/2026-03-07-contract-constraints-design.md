# 支线合约进阶设计（Time Limit / Stealth）

## 背景

当前项目已经具备最小可玩的支线合约系统：

- 合约目标支持击杀 / 收集
- 合约有进度、完成与奖励
- UI 侧栏已有基础合约面板
- 运行时已拆分为 `contracts` / `ai` / `snapshot` 等模块，便于继续扩展

下一批功能需要在不破坏现有主线流程的前提下，为支线合约增加更丰富的约束玩法：

- `time_limit`：限定回合内完成
- `stealth`：在不暴露的情况下完成

本轮目标是做出一套可扩展、可测试、可维护的“通用合约约束层”，而不是分别硬编码两套特例逻辑。

## 目标

本轮实现应满足以下目标：

1. 为支线合约新增约束层，支持无约束、单约束和双约束组合。
2. 支持两类新约束：
   - `TimeLimit { max_turns }`
   - `Stealth`
3. 在合约面板与日志中清晰展示约束状态、成功与失败结果。
4. 失败的支线合约不影响主线通关，但不会继续累计进度，也不会发放奖励。
5. 运行时实现应尽量集中在 `contracts.rs`，避免让 `Game` 主文件再次膨胀。

## 非目标

本轮不做以下内容：

- 不改主线胜利条件。
- 不引入完整潜行系统（例如可见度、独立仇恨值、听觉锥等）。
- 不改怪物 AI 的核心决策结构，只利用现有 `Alert` 状态作为 stealth 失败信号。
- 不做复杂的合约配置表或外部数据驱动生成器。
- 不一次性扩展到所有目标类型；本轮优先保证 `CollectItem` 目标在新约束下完整可玩。

## 设计概览

### 核心思路

将支线合约拆为两层：

- **目标（objective）**：玩家需要完成什么
- **约束（constraints）**：玩家在什么条件下完成才算有效

约束层与目标层解耦后，可以让未来更多目标类型复用同一套时限 / 潜行 / 其他约束框架，而不会出现逻辑分散、UI 不一致、测试难以维护的问题。

### 推荐方案

采用**通用合约约束层**方案：

- 为 `SideContract` 增加约束集合与失败状态
- 在 `contracts.rs` 内集中管理失败判定、进度冻结、日志与奖励发放
- 在 `snapshot.rs` 为 UI 暴露结构化约束视图
- 在 AI/回合推进处只负责发出有限的“状态事件”

## 数据模型

### 目标层

保留现有 `ContractObjective` 体系，继续沿用当前已验证的目标逻辑。

本轮重点支持：

- `CollectItem`

兼容已有：

- `KillMonsters`

### 约束层

新增 `ContractConstraint`：

- `TimeLimit`
  - 记录 `start_turn`
  - 记录 `max_turns`
  - 允许派生剩余回合显示
- `Stealth`
  - 记录是否已暴露

### 合约状态

`SideContract` 在现有 `completed` 基础上补充：

- `constraints: Vec<ContractConstraint>`
- `failed: bool`
- `failure_reason: Option<String>`

语义定义：

- `completed == true`：已完成且可领奖（或已领奖）
- `failed == true`：约束被破坏，合约失败
- `completed` 与 `failed` 均为终态，不再反复变化

## 规则定义

### Time Limit

#### 规则

- 合约生成时记录 `start_turn`
- 剩余回合计算：

`remaining = max_turns - (current_turn - start_turn)`

- 当剩余回合小于 0 时，合约立即失败
- 若在超时前已完成目标，则正常完成并发放奖励

#### UI 表现

- 在合约面板显示：`剩余: N 回合`
- 剩余回合较低时，日志提示一次即可，避免刷屏

### Stealth

#### 规则

本轮采用最小、可测、与现有系统一致的定义：

- 只要任意怪物进入 `Alert`，当前带有 stealth 约束的支线合约立即失败

这样可以直接复用现有：

- 视野追击
- 噪音触发警觉
- `MonsterAiState::Alert`

无需新增独立潜行系统。

#### UI 表现

- `潜行: 未暴露`
- `潜行: 已失败`

## 成功 / 失败语义

### 成功

当以下条件同时满足时，支线合约成功：

1. 目标完成
2. 所有约束仍处于满足状态

### 失败

任一约束破坏时：

- 立即切换为失败态
- 记录失败原因
- 写入日志
- 停止后续累计进度
- 停止奖励发放

### 与主线关系

失败支线：

- 不阻止主线通关
- 仍在面板中展示失败结果，便于玩家理解发生了什么

## 运行时职责划分

### `contracts.rs`

作为支线合约系统的核心模块，负责：

- 合约约束状态检查
- 合约失败处理
- 合约进度冻结
- 完成时奖励发放
- 与 UI 快照所需的状态文本整理

建议新增集中方法：

- `tick_contract_constraints(turn)`
- `on_contract_alert_triggered()`
- `fail_side_contract(reason)`
- `can_progress_side_contract()`

### `ai.rs`

负责在怪物从非 `Alert` 转入 `Alert` 时，通知合约系统 stealth 约束可能失败。

### `actions.rs` / 回合推进

在玩家行动导致回合前进后，触发限时约束检查。

### `snapshot.rs`

为 UI 暴露结构化合约视图：

- 合约状态
- 约束文本
- 失败文案

## UI 设计

在现有侧栏合约面板上增强信息密度，但不引入复杂布局重写。

推荐展示顺序：

1. 名称
2. 目标
3. 进度
4. 约束列表
   - `剩余: 8 回合`
   - `潜行: 未暴露`
5. 状态
   - `进行中`
   - `已完成`
   - `已失败`

失败时面板要能看到失败结果，不依赖日志回看。

## 生成策略

本轮使用“最小可玩样本”策略，不引入复杂配置：

- 某些 `CollectItem` 合约附带 `TimeLimit`
- 某些 `CollectItem` 合约附带 `Stealth`
- 少量概率附带双约束

后续如果需要，可再将生成逻辑外提为配置驱动。

## 存档兼容

由于 `SideContract` 会新增字段，本轮需要纳入序列化。

兼容策略：

- 为新增字段提供 `serde(default)`
- 旧存档加载时，默认视为“无约束、未失败”

这样可以避免因为结构扩展导致旧存档直接失效。

## 测试策略

### 合约模块测试

需要覆盖：

- 限时合约在超时前完成可成功
- 限时合约超时后失败
- stealth 合约在怪物进入 `Alert` 后失败
- 已失败合约不再累计进度
- 已失败合约不再发奖励

### 快照 / UI 视图测试

需要覆盖：

- 合约视图包含剩余回合
- 合约视图包含潜行状态
- 合约视图正确展示失败状态与失败文案

### AI 联动测试

如有必要，增加“进入 `Alert` 触发 stealth 失败”的回归测试。

## 风险与控制

### 风险 1：状态散落

如果在 `mod.rs`、`ai.rs`、`actions.rs` 同时各写一部分失败逻辑，后续维护会变差。

**控制方式：** 所有合约状态变更集中走 `contracts.rs`。

### 风险 2：日志与面板不一致

如果日志使用一套文案、面板使用另一套状态来源，容易出现显示错位。

**控制方式：** 由合约状态统一驱动 snapshot 视图。

### 风险 3：旧存档破坏

**控制方式：** 通过 `serde(default)` 给新字段补默认值。

## 分阶段落地建议

建议按以下顺序实现：

1. 扩展数据结构与序列化兼容
2. 为 `contracts.rs` 加失败态与约束检查方法
3. 接入 `time_limit`
4. 接入 `stealth`
5. 增强 snapshot / UI 面板
6. 完成回归测试与联动测试

## 结论

本轮采用“通用合约约束层”方案，可以在不大幅改动现有玩法骨架的前提下，完成：

- `time_limit`
- `stealth`
- 双约束组合
- 失败态展示
- 可维护的后续扩展基础

这是当前代码结构下，风险最低、维护性最好的实现路线。
