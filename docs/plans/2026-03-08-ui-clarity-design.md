# UI 可读性改进设计（地图分类配色 / 背包分组）

日期：2026-03-08

## 背景

当前项目已经有稳定的 `ratatui` UI 框架、地图快照渲染、背包弹窗和基于物品定义的数据驱动系统，但游玩体验里存在两个明显的可读性问题：

- 地图中的角色、敌人、物品、门、陷阱目前主要依赖字符区分，颜色几乎只体现“可见 / 已探索 / 隐藏”，导致识别成本高。
- 背包弹窗把所有物品按单一列表平铺展示，虽然有属性说明和操作提示，但实际选择时不够直观，尤其在装备、消耗品、任务物混在一起时更难快速定位。

本次目标是在不改变核心玩法逻辑的前提下，提升地图与背包 UI 的可读性，让玩家更容易“扫一眼就知道哪里危险、哪里可拾取、当前能操作什么”。

## 目标

本批实现以下最小闭环：

1. 地图对当前可见对象按类别上色，而不是统一白色。
2. 已探索但当前不可见区域继续保持低饱和度，避免画面过花。
3. 背包按装备槽位/物品类别分组展示，而不是单一长列表。
4. 当前选中项提供明显高亮，并强化“当前可操作行为”的提示。
5. 不改变现有背包操作按键与底层物品逻辑。
6. 补充必要测试，保证快照映射和 UI 文本结构稳定。

## 非目标

本批不做：

- 新增背包排序热键、分页或过滤输入
- 新增地图图例弹窗或主题切换
- 修改物品使用、装备、卸下、丢弃的底层规则
- 调整地图生成、怪物行为或战斗数值
- 引入复杂的颜色配置文件系统

## 方案选择

本批采用**方案 A：最小展示层增强**。

原因：

- 痛点集中在“看不清”和“不好选”，优先改展示层即可直接改善体验。
- 现有数据模型已经能区分大多数对象类型，不需要为此重写背包或地图逻辑。
- 把改动集中在 `snapshot` 与 `ui` 层，能降低回归风险，也更容易通过测试锁定行为。

未采用的方案：

- 更激进的背包交互重做（分页、左右切栏、过滤器）会增加学习成本。
- 仅做少量颜色微调不能解决背包混排的核心问题。

## 地图配色设计

### 基本原则

地图显示采用“两层语义”：

- 第一层是可见性：隐藏 / 已探索 / 当前可见
- 第二层是对象类别：玩家 / 敌人 / 地面物品 / 门 / 陷阱 / 墙 / 地板

其中：

- `Hidden`：保持黑色或空白
- `Explored`：统一使用暗灰，继续弱化历史区域
- `Visible`：根据对象类别着色

这样既能维持现有 FOV 的信息层次，又能在当前视野里快速区分对象。

### 建议颜色映射

- 玩家 `@`：亮青色或亮绿色，作为最强视觉锚点
- 敌人：亮红色，强调威胁
- 地面物品：亮黄色，强调“可拾取”
- 门：黄色或深黄色，突出可交互性
- 陷阱 `^`：洋红或紫红，强调危险
- 墙：浅灰
- 地板：深灰或白灰

颜色选择不追求“完全写实”，优先保证终端中的识别度。

### 数据落点

当前 `MapCell` 只有 `ch` 和 `tone`，不足以表达类别信息。

建议新增一个简单的显示类别枚举，例如：

- `Player`
- `Monster`
- `Item`
- `Door`
- `Trap`
- `Wall`
- `Floor`
- `Unknown`

`snapshot` 层负责把地图和实体状态映射成这个类别；`ui` 层只负责根据类别 + tone 决定颜色样式。

这样可以把“对象识别”与“如何渲染”解耦，后续要再调色时不需要回改游戏逻辑。

## 背包分组设计

### 分组规则

背包显示从“单列表”改为“按装备槽位 / 用途分组”。建议固定顺序：

1. 武器
2. 护甲
3. 饰品
4. 消耗品
5. 任务物
6. 其他

判断依据：

- `equipment(slot = weapon)` -> 武器
- `equipment(slot = armor)` -> 护甲
- `equipment(slot = accessory)` -> 饰品
- `consumable` / `buff_consumable` -> 消耗品
- `quest_package` / `quest_item` -> 任务物
- 未识别类型 -> 其他

空分组默认不显示，避免背包弹窗被空标题撑长。

### 交互规则

保持现有按键不变：

- `w/s` 选择上一项 / 下一项
- `Enter` 使用或装备
- `r` 卸下
- `x` 丢弃
- `i` / `Esc` 关闭背包

也就是说，本次只改变“如何展示和提示”，不改变“如何操作”。

### 高亮规则

当前选中项做两层强化：

- 选中行使用明显高亮样式，例如底色、反色或亮白粗体
- 行首保留 `>` 指示符，避免只靠颜色传达焦点

同时，把“当前项可执行什么”放到主行上，以短标签形式直观显示，例如：

- `可装备`
- `可使用`
- `可卸下`
- `可丢弃`
- `不可操作`
- `任务物`

已装备条目继续保留 `[已装备]` 标记，但应出现在主行的高关注位置，而不是淹没在次级说明中。

### 文本层级

为了让选择更快，背包每条物品展示拆成两层：

- 主行：名称、数量、已装备状态、操作标签
- 次行：属性说明 `attr_desc`

主行承担“快速决策”，次行承担“补充说明”。

## 模块落点

- `src/game/snapshot.rs`
  - 为地图快照补充显示类别
  - 为背包条目补充分组信息与操作标签文本
- `src/game/ui/mod.rs`
  - 按地图类别决定颜色样式
  - 按分组渲染背包标题与条目高亮
- `src/game/inventory.rs`
  - 原则上不改业务逻辑
  - 仅在测试或展示辅助确有需要时再最小调整
- `docs/project-progress.md`
  - 实现完成后补充本次 UI 可读性升级记录

## 测试设计

本批重点测试展示映射，而不是重复测试既有背包业务逻辑。

建议覆盖：

1. 地图快照能把玩家、敌人、物品、门、陷阱映射为正确显示类别。
2. 背包条目能根据物品效果/槽位归到正确分组。
3. 背包条目的操作标签符合当前状态，例如已装备装备应显示可卸下而不是可丢弃。
4. UI 渲染辅助函数能为不同地图类别返回不同颜色样式。
5. 背包分组渲染结果中，标题顺序与条目归属稳定。

## 风险与约束

- 终端配色能力有限，颜色选择必须兼顾深色终端可读性。
- 当前 `MapCell` 和 `InventoryItemView` 是快照层公共结构，新增字段后要同步更新测试构造。
- 背包分组只是展示层重排，必须确保 `inventory_selected` 与实际操作目标仍一致。

## 验收标准

- 当前可见地图中，玩家、敌人、物品、门、陷阱一眼可分。
- 已探索但不可见区域依旧低调，不影响主视野辨识。
- 背包不再把所有物品混为一列，而是能按组快速扫视。
- 当前选中项和可操作行为足够醒目，减少误选和反复阅读。
- 既有背包使用/装备/卸下/丢弃行为不回归。
