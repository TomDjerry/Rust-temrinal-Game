# 环境交互设计（门 / 陷阱 / 基础噪音）

日期：2026-03-08

## 背景

当前项目已经具备：
- 稳定的回合制移动与战斗循环
- 基于地图/FOV 的可见性系统
- AI 的 `Patrol / Alert / Flee` 状态机
- 已有的噪音事件链路与支线潜行失败联动
- 已完成的运行时模块拆分，便于在 `map / actions / ai / snapshot / ui` 中增量扩展

下一批功能选择“环境交互”，目标是在不重写主循环的前提下，为地图加入可操作对象与危险地块，提升潜行、路径选择与局内叙事。

## 目标

本批实现以下最小闭环：

1. 门：支持开/关两种状态。
2. 关闭门阻挡移动与视线；开启门允许通过且不阻挡视线。
3. 玩家撞上关闭门时可执行开门交互。
4. 新增显式关门动作，允许玩家关闭相邻已开启的门。
5. 陷阱：固定伤害，触发一次后失效。
6. 陷阱触发时制造显著噪音，沿用现有 AI 警戒机制。
7. UI/快照中可显示门与陷阱状态。
8. 保持主线胜利条件、存档兼容思路与支线约束系统不被破坏。

## 非目标

本批不做：
- 锁与钥匙
- 怪物主动开关门
- 隐藏陷阱、侦测陷阱、拆除陷阱
- 更复杂的声学传播或声音衰减模型
- 多门型/多陷阱型的配置系统

## 方案选择

本批采用**方案 A：最小环境层**。

原因：
- 变更集中，能复用现有地图、行动、AI、快照和 UI 模块。
- 不引入新的通用环境交互抽象，避免在对象种类尚少时过度设计。
- 能最快形成“门控路径 + 陷阱风险 + 噪音惊动怪物”的可玩闭环。

## 数据模型设计

### 门

在地图层为门增加显式状态，建议通过地图格类型或同等简单表示完成：
- `ClosedDoor`
- `OpenDoor`

语义：
- `ClosedDoor`：阻挡移动、阻挡视线
- `OpenDoor`：可通行、不阻挡视线

本批不额外引入门对象列表，优先复用现有地图格系统，降低同步复杂度。

### 陷阱

新增独立陷阱列表，例如：
- 位置 `pos`
- 固定伤害 `damage`
- 是否已触发 `triggered`

原因：
- 与门不同，陷阱更像“覆盖在地板上的一次性环境效果”
- 保持地图主格仍然是地板，避免把陷阱触发状态编码到主地形枚举里导致分支膨胀

## 规则设计

### 开门

- 玩家尝试移动到关闭门格时，不直接进入目标格。
- 本回合改为执行“开门”交互。
- 开门消耗一回合。
- 开门产生中等噪音。

### 关门

- 新增一个显式动作，例如使用单键触发。
- 尝试关闭玩家四邻接中的第一扇开启门。
- 成功时：门变为关闭、消耗一回合、产生中等噪音。
- 失败时：仅记录日志，不消耗回合。

### 陷阱

- 玩家进入带陷阱的位置时立即触发。
- 造成固定伤害。
- 触发后标记为 `triggered = true`，之后不再重复造成伤害。
- 陷阱触发时产生高噪音。
- 陷阱伤害允许导致死亡。

### 噪音

继续复用现有噪音与 AI 警戒链路，不新做独立系统。
新增三类来源：
- 开门
- 关门
- 陷阱触发

约束：
- 噪音强度只区分相对等级即可，不必引入复杂配置表
- 只要怪物因噪音进入 `Alert`，现有 stealth 合约失败逻辑自然生效

## 地图生成设计

采用最小生成策略：
- 地图上生成少量关闭门
- 地图上生成少量陷阱
- 不追求复杂模板化，只追求稳定出现与基础可玩性

门的最小落点建议：
- 放在房间与走廊的连接处，或走廊中的窄口位置

陷阱的最小落点建议：
- 放在可达地板上，避开玩家出生点、出口点和关键任务物初始格

## 模块落点

- `src/game/map/`
  - 增加门相关地图判定与陷阱数据
  - FOV/阻挡判定识别关闭门
- `src/game/actions.rs`
  - 撞门开门
  - 显式关门动作
  - 移动后触发陷阱
- `src/game/ai.rs`
  - 继续复用噪音响应，不引入门专属 AI 行为
- `src/game/snapshot.rs`
  - 地图快照输出门/陷阱可视信息
- `src/game/ui/mod.rs`
  - 门/陷阱符号显示
- `src/game/mod.rs`
  - 仅在必要时放少量核心类型，不堆业务逻辑

## UI 设计

地图显示遵循最小可读原则：
- 关闭门：与墙/地板明显区分的字符
- 开启门：与关闭门不同字符
- 未触发陷阱：可见符号
- 已触发陷阱：弱化符号或还原为普通地板表现（实现时二选一即可）

日志至少包含：
- 你打开了门
- 你关上了门
- 你触发了陷阱，受到 X 点伤害

## 存档与兼容

如果新增门状态或陷阱列表进入存档：
- 应继续通过 `serde(default)` 或同等策略保持旧存档兼容
- 旧存档缺失环境字段时，默认回落为空门/空陷阱数据或从现有地图状态恢复

## 测试策略

优先补以下回归测试：

1. 关闭门阻挡移动。
2. 撞门会打开门并消耗回合。
3. 相邻开启门可被关闭；无可关闭门时不消耗回合。
4. 关闭门阻挡 FOV，开启门不阻挡 FOV。
5. 陷阱只触发一次。
6. 陷阱触发造成伤害并制造噪音。
7. 门/陷阱在快照中正确显示。
8. 噪音触发下的 AI 警戒与 stealth 失败链路不回归。

## 风险与控制

### 风险 1：地图/FOV/移动判定不一致

控制：
- 所有“是否阻挡移动/视线”的判断尽量复用同一来源
- 为门单独补阻挡与 FOV 测试

### 风险 2：环境交互导致主循环膨胀

控制：
- 行为逻辑主要放到 `actions.rs` 与 `map/` 相关模块中
- `mod.rs` 只保留必要的核心类型与初始化 wiring

### 风险 3：噪音联动引入 stealth 回归

控制：
- 复用现有噪音事件机制
- 增补“门/陷阱噪音 -> Alert -> stealth fail”定向测试

## 结论

本批环境交互采用“门 + 固定伤害陷阱 + 基础噪音”的最小方案，优先形成稳固、可测试、可扩展的环境层。后续若继续扩展锁、钥匙、隐藏陷阱或怪物开门，可在本批的数据和行动边界上平滑演进。
